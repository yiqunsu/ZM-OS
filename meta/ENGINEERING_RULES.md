# FilmOS 工程规范

状态：Normative

本文件约束非临时性的工程做法。具体框架版本规则以对应目录中的 `AGENTS.md` 和官方文档为补充。

## 1. 总体原则

- 修改前必须先确认工作区状态，保留用户已有的未提交变更。
- 优先做满足需求的最小完整改动，不借机重写无关模块。
- 业务正确性优先于界面上的乐观效果。
- 一个用户动作涉及多条数据时，必须先设计清楚事务边界和失败恢复方式。
- 不得为了让检查通过而删除有效测试、降低关键校验或吞掉异常。
- 新规则应尽可能由类型、Schema、数据库约束和测试执行。

## 2. 后端边界

### Router

- Router 只负责 HTTP 入参、依赖注入、鉴权、调用用例和响应模型。
- Router 不应该包含查询拼装、状态流转或跨实体业务逻辑。
- 部分更新使用清晰一致的语义；新增接口时优先区分完整替换与部分更新。

### Service

- Service 负责业务规则、授权后的用例执行和事务边界。
- 一个业务用例应一次提交；需要组合的 Service 不应在中间步骤提前 `commit`。
- 多实体操作必须在同一个数据库事务中完成。
- Service 不得信任前端已经验证过的数据。
- 对可预期冲突返回稳定的业务错误，不能把外键或唯一约束异常直接暴露成通用 500。

### Repository

- 当前项目不强制增加 Repository 层。
- 当查询复杂度、复用度或数据源数量显著增加时，再将纯数据访问从 Service 下沉。
- 不得仅为满足分层形式而创建只转发一行 SQLAlchemy 调用的 Repository。

## 3. API 与 Schema

- API JSON 字段统一使用 `snake_case`。
- 请求 Schema 必须表达基本约束，例如正数、枚举、最小列表长度和字符串长度。
- 错误响应应有稳定错误码和面向用户的中文信息；前端不应依赖解析任意错误文本判断逻辑。
- 列表规模会持续增长的资源必须设计服务端筛选和分页，不能长期依赖前端加载全表。
- 修改 API 契约时应同步更新前端类型、调用方、测试和相关文档。
- 避免在多个前端组件手写重复的后端实体类型；条件成熟时从 OpenAPI 生成类型。

## 4. 数据库

- 所有业务表结构变更必须通过 Alembic migration。
- migration 必须能从空数据库顺序升级到 `head`，不能只在已有开发库上验证。
- 外键、唯一性、非空、取值范围和关键状态约束应尽可能下沉到数据库。
- PostgreSQL 不会自动为所有外键创建索引；必须根据查询路径显式评估索引。
- 涉及数量、金额或要求精确比较的数值优先使用 `Decimal/Numeric`，避免无意识使用浮点数。
- 时间统一保存为带时区时间；涉及营业日期时必须使用已确认的业务时区。
- 历史业务记录优先归档或停用，不能默认物理删除。
- 迁移和批量修复脚本必须可审查、可重复验证，并明确回滚或恢复策略。

## 5. 前端

- 前端不得直连数据库；业务数据统一通过 `lib/api.ts` 或后续统一 API 层访问。
- 前端可以做交互校验，但不能成为业务规则的唯一执行者。
- 一次需要原子性的用户操作不得由前端拼接多个相互竞争的写请求实现。
- 乐观更新失败时必须恢复权威服务器状态，并向用户显示明确反馈。
- 禁止静默吞掉关键加载或写操作错误。
- 加载、空状态、失败状态和重试入口应保持一致。
- 页面或组件承担列表、表单、多个弹窗和数据访问等多种职责时，应按 feature、hook 和子组件拆分。
- 通用领域类型应集中维护，不应在多个页面重复声明近似接口。
- 新增依赖前先判断现有能力是否足够，并同步更新 lockfile。
- 修改 Next.js 代码前必须遵守 `frontend/AGENTS.md`，并查看当前安装版本对应的本地官方文档。

## 6. 状态与并发

- 订单和生产任务状态只能通过明确的业务动作迁移，不能用任意字段更新绕过规则。
- 并发创建编号、队列位置或占用订单时，必须使用数据库锁、序列、约束重试或其他可靠策略。
- 唯一约束是最后防线，不代替友好的冲突处理。
- 重排队列应由单个批量后端用例完成，并校验任务集合和目标机器。

## 7. 鉴权与安全

- 生产环境不得使用默认密钥、默认数据库密码或宽泛的 CORS 配置。
- 配置缺失或仍为不安全默认值时，生产进程应该快速失败。
- 后端必须执行权限判断；隐藏前端按钮不等于授权。
- 日志和错误信息不得包含密码、Token、Cookie、完整密钥或不必要的客户敏感数据。
- 新增外部输入时，必须考虑认证、授权、输入限制和滥用风险。

## 8. 日志与审计

- 应用日志输出结构化 JSON 到 stdout，由 Docker 收集。
- HTTP 请求日志应包含时间、级别、方法、路径、状态码、耗时和 request ID。
- 关键业务写操作应记录操作者、动作、对象、结果和必要的前后差异。
- 运行日志不能代替长期业务审计；业务审计不能只存在于容器日志。
- Sentry 负责异常与性能问题；Phoenix 负责 AI Agent Trace；两者不能替代结构化运行日志或业务审计。
- Phoenix 埋点必须与业务代码解耦，并由配置开关控制；采集端故障不得阻断请求或改变业务结果。
- 生产环境启用 Phoenix 前，必须配置访问控制、传输安全、数据保留策略和固定镜像版本。
- Prompt、模型回复和工具参数可能进入 Trace，不得把 Phoenix UI 作为无保护的公网服务暴露。
- Docker 服务必须配置合理的日志轮转，防止磁盘无限增长。

详细边界和操作方式见 `meta/OBSERVABILITY.md`。

## 9. 代码质量

- Python 新代码保持类型标注并通过 Ruff；TypeScript 保持 strict 模式。
- 命名应使用领域语言，避免 `data`、`handleThing` 等无法表达含义的抽象。
- 注释解释“为什么”，不要逐行复述代码。
- 不保留失效的兼容代码、注释掉的大段实现或无用途依赖。
- 发现代码与本目录文档冲突时，应在本次变更中修复，或明确记录为后续事项。

## 10. Definition of Done

一个变更完成至少意味着：

- 需求和业务不变量得到满足；
- 失败路径和并发影响已考虑；
- 相关测试已新增或更新；
- lint、测试和构建按 `TESTING.md` 通过；
- migration、配置、文档和示例环境变量已同步；
- 没有覆盖用户无关改动，没有提交秘密；
- 交付说明包含改动结果、验证结果和仍存在的限制。
