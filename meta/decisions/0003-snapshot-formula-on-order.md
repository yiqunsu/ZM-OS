# 0003 订单保存配方快照

状态：Accepted

日期：2026-07-08

## 背景

配方属于可持续维护的基础数据。同一配方以后可能修改名称、规格或材料，但历史订单必须保留当时实际采用的信息，不能随着基础配方变化而变化。

## 决策驱动因素

- 历史订单需要可追溯；
- 生产记录不能因基础数据编辑而被改写；
- 订单仍需要知道它来源于哪个配方；
- 快照读取应简单可靠。

## 考虑过的方案

### 订单只保存 `formula_id`

结构简单，但修改配方会追溯改变历史订单展示，无法可靠还原当时信息。

### 配方版本表

版本能力完整，但对当前项目复杂度较高，需要处理版本发布、引用和迁移。

### 订单保存配方引用和快照

保留来源关系，同时冻结订单使用的信息，适合当前阶段。

## 决策

- 订单可以保存 `formula_id` 作为来源引用；
- 选择配方时，将配方名称、规格参数和材料复制到订单的 `formula_snapshot`；
- 后续修改基础配方不得改变已有订单快照；
- 显式更换订单配方时，为该订单生成新的快照。

## 结果与代价

### 正面影响

- 历史订单稳定且可追溯；
- 读取订单详情无需查询配方历史版本；
- 基础配方可以继续演进。

### 负面影响

- 配方数据会在订单中重复存储；
- 快照结构需要保持兼容；
- 必须明确哪些字段属于快照。

### 后续工作

- 统一快照字段命名为 API 约定的 `snake_case`；
- 为快照生成和基础配方更新后的不变性保留自动化测试；
- 如果未来需要完整版本比较，再评估正式配方版本模型。
