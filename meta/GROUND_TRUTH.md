# FilmOS Ground Truth

状态：Normative

本文件记录 FilmOS 的产品边界、领域语言和目标业务不变量。除“待确认事项”外，下面的规则应被视为系统必须长期保持的事实。

## 1. 产品目标与范围

FilmOS 是面向塑料薄膜工厂的订单与生产协同系统，用于替代“微信接单、人工录入 Excel、白板排产”的工作方式。

当前核心闭环是：

1. 维护客户、产品、产品大类、配方、机器等基础数据；
2. 手工或通过 AI 助手录入订单；
3. 待排订单进入排单看板；
4. 订单被分配到机器上的生产任务；
5. 生产任务完成后，关联订单同步完成。

系统当前不是财务、库存、采购、仓储或完整 ERP。新增这些领域前应先明确边界和数据归属。

## 2. 系统边界与数据归属

- FastAPI 后端是唯一业务入口和业务规则执行者。
- Next.js 前端负责展示与交互，不直接连接数据库，也不独立实现权威业务规则。
- PostgreSQL 是业务数据的权威来源。
- Alembic 是业务数据库结构演进的权威方式。
- Redis 目前不是任何核心业务数据的权威来源；没有明确用途时不得把业务正确性依赖于 Redis。
- Phoenix 只保存 AI Agent 的可观测性 Trace，不是业务数据库、业务审计、应用日志或错误追踪的权威来源。
- Phoenix 必须是可关闭的旁路能力；关闭、不可用或初始化失败时，订单、排产和其他业务 API 必须继续工作。
- Phoenix 数据使用独立存储，不得写入 FilmOS 业务表或影响 PostgreSQL 事务。
- AI 助手与普通页面必须复用同一套后端业务能力，不能形成两套不同规则。
- AI 发起建单、改单、排产等写操作时，必须经过用户明确确认后才执行。

## 3. 领域术语

### 订单（Order）

客户对某个产品的一次生产需求，包含规格参数、数量、单位、配方选择和备注。

订单状态：

- `PENDING`：待排单；
- `PRODUCING`：已进入生产任务；
- `DONE`：生产已完成。

### 生产任务（ProductionTask）

机器队列中的一个生产单元。一项任务可以包含一张或多张可合并生产的订单，并在机器队列中拥有顺序位置。

任务状态：

- `WAITING`：等待生产；
- `PRODUCING`：生产中；
- `DONE`：已完成。

### 配方与配方快照（Formula / Formula Snapshot）

配方是可维护的基础数据；配方快照是订单当时采用的历史生产信息，属于订单自身。

### 基础数据（Master Data）

客户、产品大类、产品、配方、机器和机型/花纹等被其他业务对象引用的长期数据。

## 4. 订单不变量

- 订单号必须唯一、稳定，已经发出的订单号不得因删除其他订单而复用。
- 订单数量必须大于零；单位必须来自系统支持的明确集合。
- 一张订单同一时间最多属于一个生产任务。
- `PENDING` 订单不得关联生产任务。
- `PRODUCING` 订单必须关联一个未完成的生产任务。
- 订单完成应由生产任务完成驱动，订单状态与任务状态不得相互矛盾。
- 将订单撤出未完成任务后，如果它没有被原子地转移到其他任务，应回到 `PENDING`。
- 删除或修改订单不得留下空任务、悬空引用或与看板不一致的状态。
- 客户、产品、数量、单位等关键字段必须由后端验证，不能只依赖前端表单。

## 5. 配方不变量

- 选择配方创建订单时，必须在订单上保存配方快照。
- 后续修改基础配方，不得追溯改变已有订单的快照。
- 显式更换订单配方时，可以基于新配方重新生成该订单的快照。
- 配方必须与订单产品兼容；不允许静默接受不存在或不匹配的配方。
- 配方被订单引用时不得直接删除；衍生配方关系不得形成自引用或无法解释的循环。

## 6. 排产与机器不变量

- 一次用户可感知的拖拽、移动、合并、拆分或重排，必须作为一个原子业务操作完成。
- 创建生产任务前，后端必须确认机器和全部订单存在且处于允许排产的状态。
- 停用机器不得接收新任务；已有历史任务不能因机器停用而丢失。
- 把订单从一个任务转移到另一个任务时，不能出现同时属于两个任务或两个任务都不包含它的中间持久状态。
- 完成任务时，其关联订单必须在同一事务中同步完成。
- 删除未完成任务时，其订单必须在同一事务中解除关联并回到待排状态。
- 机器的产品大类、宽度范围、花纹能力是排产兼容性数据。具体哪些条件属于硬限制、哪些只做提示，见“待确认事项”。

## 7. 基础数据不变量

- 已被业务数据引用的客户、产品、配方和机器不得直接物理删除，除非有明确迁移或归档策略。
- 基础数据的删除保护必须由后端和数据库共同保证。
- 是否允许客户、产品、配方或机器重名，需要按“待确认事项”处理，不能由实现自行猜测。

## 8. 用户与安全

- 所有业务 API 默认需要认证，登录接口和健康检查等明确公开接口除外。
- 密码、JWT、Cookie、API Key 和数据库凭证不得写入日志或提交到 Git。
- Agent Trace 可能包含 Prompt、回复和工具参数，应按可能包含客户业务信息的数据管理，限制访问并设置保留策略。
- `OWNER` 与 `OPERATOR` 是不同角色；权限矩阵尚未完全确定前，不得假设两者永久拥有相同权限。

## 9. 待确认事项

以下问题尚未形成 Ground Truth。实现涉及它们时必须先确认：

1. FilmOS 的正式业务时区是否固定为 `Asia/Shanghai`，以及订单号日期是否按该时区生成；
2. `WAITING` 与 `PRODUCING` 的准确切换方式，是人工开始、自动开始还是队首任务自动生产；
3. 已完成任务和订单能否重新打开，若能，允许哪些角色执行；
4. 机器宽度、产品大类和花纹能力是强制限制还是仅提示；
5. 可合并生产订单的完整判定规则；
6. `OWNER` 与 `OPERATOR` 的最终权限矩阵；
7. 基础数据是否允许重名以及归档策略；
8. 数量单位是否只允许 `kg`、`t`，以及是否还需要卷、米等单位。
