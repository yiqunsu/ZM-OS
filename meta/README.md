# FilmOS 项目长期记忆

`meta/` 保存需要跨功能、跨会话长期保持一致的项目知识。它服务于产品负责人、开发者和编程 Agent，不代替面向使用者的项目 README，也不代替代码、测试和数据库约束。

## 文档地图

| 文档 | 用途 | 何时阅读 |
| --- | --- | --- |
| [GROUND_TRUTH.md](GROUND_TRUTH.md) | 产品边界、领域术语和业务不变量 | 修改业务流程、模型、状态或 API 前 |
| [ENGINEERING_RULES.md](ENGINEERING_RULES.md) | 前后端、数据库、安全和可维护性规范 | 修改任何代码前按涉及范围阅读 |
| [TESTING.md](TESTING.md) | 最低验证要求和常用命令 | 开始实现及交付前 |
| [OBSERVABILITY.md](OBSERVABILITY.md) | 日志、Sentry、Phoenix 与业务审计的职责边界 | 修改可观测性、追踪或部署配置前 |
| [decisions/](decisions/) | 重要决策的背景、选择和后果 | 改变架构或长期约定前 |

## 规则强度

本文档统一使用以下措辞：

- **必须 / 禁止**：不可自行偏离。确需改变时，应同步修改文档、测试及相关 ADR。
- **应该 / 不应该**：默认遵守；偏离时需在变更说明中解释原因。
- **可以**：可选做法。
- **待确认**：尚未形成项目事实，不得由开发者或 Agent 擅自假设。

## 信息优先级

发生冲突时按以下顺序处理：

1. 当前任务中产品负责人的明确要求；
2. `GROUND_TRUTH.md` 中的业务不变量；
3. 状态为 `Accepted` 的 ADR；
4. `ENGINEERING_RULES.md` 与 `TESTING.md`；
5. README 和代码注释。

如果代码与 Ground Truth 冲突，应将其视为待修复的实现偏差，而不是静默修改文档来迎合代码。涉及真实业务含义且无法从现有资料确认时，应先向产品负责人确认。

## 维护方式

- 文档与相关代码放在同一个变更中提交。
- 新增或改变业务不变量时，同时更新 Ground Truth、自动化测试和必要的数据库约束。
- 重要且长期的技术选择使用 ADR；局部实现细节不需要 ADR。
- ADR 被新决策取代时保留原文件，并标记为 `Superseded`。
- 能由类型、lint、测试或数据库约束执行的规则，不应只停留在 Markdown。
- 避免在多份文档复制同一段规则；本文件只负责导航。
